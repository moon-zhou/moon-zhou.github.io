签到类需求的设计与思考

2019-08-12

背景

6月底，所在会员研发中心接到原始需求，需要针对特定的频道开发签到功能。
基本要求：
    1. 各频道签到页面可以动态配置
    2. 各频道签到规则可不一致
    3. 签到有每日奖品及累计奖品发放
    4. 周期内可以通过补签卡进行补签
    5. 补签和签到效果一致，发放对应奖品

前期需求的讨论

  1. 动态配置页面的实现
    • 了解现有的内容配置系统，看是否满足要求
    • 确认现有内容管理不能完全满足，内容系统制管理内容
    • 需要一个布局系统，整合内容管理的展示
  2. 签到奖励是通过谁来发送（功能落到哪个系统）
    • 签到数据在签到系统
    • rpc调用任务平台发送签到奖励和补签奖励
  3. 补签卡的发放和使用
    • 完成相关任务，不是直接调用券系统进行发放，而是通过任务平台–》礼包系统–》券系统
  4. 补签用卡
    • 直接调用券系统

架构设计

  1. 后台配置管理系统
    • 布局管理（和内容管理系统有关）
    • 独立的文案配置（本质上也可以放到内容管理系统，但产品认为配置的任务文案，方便统一查看）
    • 签到信息的查询和导出
    • 补偿数据的查询
    • 分布式job的实现：用户配置推送信息的job（上一篇全表扫描），补偿数据的job
  2. 服务系统的设计
    • 开放签到等相关接口的rpc服务
    • rpc接口不局限于用户操作业务相关的，还包含job服务相关接口
  3. 通过网关进行包装
    • rpc只能是系统内网之间的通信服务，外部小程序或者H5只能通过http进行访问，需要金融网关对其进行一层包装，将rpc接口转换为http接口。

后端系统功能

  1. 整体采用springboot，包含两个代码库：springboot、vue
  2. vue代码库的最终打包文件放到springboot里
  3. 搭建过程可参考之前springboot+vue环境搭建
  4. 正常业务：频道管理，页面布局管理，以及补偿数据查询
  5. job入口：用户推送消息job，签到失败补偿job（配置的到分布式任务平台，周期性调用）

服务端系统功能

  1. 签到类接口
    • 查询签到信息接口（签到周期，每个用户展示的周期根据其实际签到进行计算，展示的不一定一样）
    • 签到接口
    • 不前接口
  2. 页面数据展示类接口
    • 页面布局内容查询接口
    • 轮播数据即可欧
  3. 用户配置类接口
    • 查询定时提醒配置
    • 修改定时提醒配置
  4. job类服务接口
    • 用户推送接口（后端系统调用该接口，再调用下游推送系统）

核心重大决策

  1. 签到接口
    • 本签到系统数据落库成功，签到成功，
    • 下游任务平台接口调用失败/超时的情况下，调用时的参数落补偿表，后台进行手动/job补偿。
    • 任务平台的接口是幂等的，相同的参数调用不会频繁发送签到奖励。
    • 任务平台调用风控时，如果认为是有风险的，即使补偿也无法成功，该情况只能等用户反馈后，由下游礼包等奖励系统，直接手动补偿。
  2. 补签接口
    • 连续补签时，任务平台没有批量接口
    • 业务上有系统，即签到系统，多线程批量调用，默认每一次单独补签为成功，调用超时/失败，补偿表进行处理
    • 补签时是不会出现风控拦截的，因为补签卡发放时已经过了一到风控，能发送补签卡可定不会被拦截补签
  3. 用户消息推送
    • 定时全表扫描（上一篇一次全表扫描XXXX）
    • 筛选符合条件的数据
    • 调用服务端接口进行推送（服务端调下游）
  4. 签到规则
    • 会员中心
      1. 周期7天内，从未签过，打开时当天即为第一天
      2. 连续签7天即为满签
      3. 一旦有一天未签，重新从第一天开始签到
    • 理财
      1. 周期7天内，从未签过，打开时当天即为第一天
      2. 七天之内均可签到，如有遗漏，在第七天结束之前均可以进行补签
      3. 前天周后，开始新的一轮，如果新一轮一直不签到，则每次打开，当天即为第一天

后期优化

  1. job–》补偿次数可以优化为动态配置，否则达到补偿次数的数据一直占据着数据库的最前面，无法补偿。
  2. 补偿数据如果一直无法补偿，可以考虑再创建一个无法补偿的表存储，避免补偿表做过多的无用数据扫描
  3. 签到规则的提取，现在是if-else写死，后期可以抽取规则，根据条件注入，调用不同的规则实现
  4. 分析各层日志，接入紫金大盘进行接口调用量，成功率的统计分析
  5. 网关上接口的调用量和返回监控，是否需要配置上流控及返回文案
  6. rsf接口关注tp99，耗时成功率等情况

思考

  1. 业务上的需求和最终实现上可能是矛盾的，需要进行互相的妥协。比如连续补签并发调用下游系统的相关决策。
  2. 保证数据的最终一致性
